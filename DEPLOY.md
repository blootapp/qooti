# Building and Releasing Qooti (Tauri)

This guide explains how to build the app and publish an update so existing users receive it via the in-app updater.

## Prerequisites

- **Node.js** 20+ and **npm**
- **Rust** (stable)
- **Tauri signing key** (you should already have the private key; the public key is in `tauri.conf.json`)

## 1. Bump the version

Update the version in **three places** (keep them in sync):

| File | Location |
|------|----------|
| `src-tauri/tauri.conf.json` | `"version": "0.1.0"` → e.g. `"0.1.1"` |
| `src-tauri/Cargo.toml` | `version = "0.1.0"` |
| `package.json` | `"version": "0.1.0"` |

Example for version `0.1.1`:

```bash
# Optional: use sed (Git Bash / WSL) or edit by hand
# src-tauri/tauri.conf.json  -> "version": "0.1.1"
# src-tauri/Cargo.toml       -> version = "0.1.1"
# package.json               -> "version": "0.1.1"
```

## 2. Build the app (Windows)

Signing is **required** for the updater. Set your private key (and optional password) in the environment, then build:

**PowerShell:**

```powershell
$env:TAURI_SIGNING_PRIVATE_KEY = "Path or content of your private key"
$env:TAURI_SIGNING_PRIVATE_KEY_PASSWORD = ""   # if the key has a password
npm run tauri build
```

**Cmd:**

```cmd
set TAURI_SIGNING_PRIVATE_KEY=Path or content of your private key
set TAURI_SIGNING_PRIVATE_KEY_PASSWORD=
npm run tauri build
```

This runs the `beforeBuildCommand` (copy assets, prepare yt-dlp, etc.) and produces:

- **NSIS installer:** `src-tauri/target/release/bundle/nsis/qooti_0.1.1_x64-setup.exe` (version in filename may vary)
- **Signature:** `src-tauri/target/release/bundle/nsis/qooti_0.1.1_x64-setup.exe.sig`
- **MSI** (if built): under `src-tauri/target/release/bundle/msi/`

### Generate `latest.json`

After a successful signed build, run (replace `v0.1.1` with your version tag):

```powershell
node scripts/generate-latest-json.js https://github.com/blootapp/qooti-releases/releases/download/v0.1.1
```

This reads the version from `tauri.conf.json`, finds the `.exe` and `.sig` in `bundle/nsis/`, and writes `latest.json` into that folder. You’ll upload that file as a release asset in the next step.

## 3. Upload to GitHub so users get the update

The app is configured to check for updates at:

`https://github.com/blootapp/qooti-releases/releases/latest/download/latest.json`

So you need to publish the new version in the **releases repo** (`blootapp/qooti-releases`).

### Option A: Manual release

1. Open **GitHub** → repo **blootapp/qooti-releases** (or your fork).
2. **Releases** → **Draft a new release**.
3. **Choose a tag:** create tag `v0.1.1` (must match the version you set in step 1).
4. **Upload assets** (all from `src-tauri/target/release/bundle/nsis/`):
   - The installer: `qooti_0.1.1_x64-setup.exe` (exact name from the folder).
   - The signature file: `qooti_0.1.1_x64-setup.exe.sig`.
   - **`latest.json`** — use the one generated by `scripts/generate-latest-json.js` (see above). If you didn’t run the script, create it manually with this structure (replace version, URL, and signature):

   ```json
   {
     "version": "0.1.1",
     "notes": "Optional release notes",
     "pub_date": "2025-01-27T12:00:00Z",
     "platforms": {
       "windows-x86_64": {
         "signature": "<paste entire content of the .sig file>",
         "url": "https://github.com/blootapp/qooti-releases/releases/download/v0.1.1/qooti_0.1.1_x64-setup.exe"
       }
     }
   }
   ```

   - `signature`: copy the **full content** of the `.sig` file (one line).
   - `url`: direct download URL of the `.exe` asset (tag and filename must match your release).

5. Upload `latest.json` as a release asset.
6. Publish the release.

Existing installs will see the update when they next check (or when your app runs its update check).

### Option B: GitHub Actions (recommended)

Use **tauri-apps/tauri-action** to build and publish from a workflow:

1. In the repo that contains the app (e.g. your main qooti repo), add **Secrets**:
   - `TAURI_SIGNING_PRIVATE_KEY`: private key content (or path in CI).
   - `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`: password if the key is protected.

2. Add a workflow that:
   - Builds the Tauri app for Windows (and optionally macOS, like your existing `build-macos.yml`).
   - Creates a release (in the same repo or in `qooti-releases`) and uploads the installer, `.sig`, and `latest.json`.

If the release is created in a **different** repo (e.g. `blootapp/qooti-releases`), you’ll need a token with repo permissions and to configure the action to push assets to that repo. The updater URL in `tauri.conf.json` must point to that repo’s `releases/latest/download/latest.json`.

## 4. Using a different GitHub repo for releases

If you host releases in another org/repo (e.g. `yourorg/qooti`), change the updater endpoint in `src-tauri/tauri.conf.json`:

```json
"plugins": {
  "updater": {
    "pubkey": "...",
    "endpoints": [
      "https://github.com/YOUR_ORG/YOUR_REPO/releases/latest/download/latest.json"
    ]
  }
}
```

Then build and publish there: same steps as above, but create the release and upload assets in that repo.

## Quick reference

| Step | Command / action |
|------|-------------------|
| Bump version | Edit `tauri.conf.json`, `Cargo.toml`, `package.json` |
| Build (signed) | `$env:TAURI_SIGNING_PRIVATE_KEY = "..."` then `npm run tauri build` |
| Artifacts | `src-tauri/target/release/bundle/nsis/*.exe` and `*.sig` |
| Publish | New release on GitHub, upload `.exe`, `.sig`, and `latest.json` |
| Updater URL | `tauri.conf.json` → `plugins.updater.endpoints[0]` |
